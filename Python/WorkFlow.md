🌊 Flow การทำงานรวมของระบบ (System Architecture)
ระบบของมึงแบ่งเป็น 2 ช่วงใหญ่ๆ คือ "ช่วงสร้างความจำ (Offline/Setup)" และ "ช่วงคุยกับบอท (Online/Runtime)"

🛠️ ช่วงที่ 1: การสร้างความจำให้บอท (Setup Phase)
ส่วนนี้ทำแค่ครั้งเดียว (หรือทำเมื่อมีการอัปเดตข้อมูลใหม่) เพื่อป้อนข้อมูลให้ระบบ RAG

เตรียมไฟล์ข้อมูล: มึงเตรียมไฟล์ .csv (เช่น thai_nutrition.csv, mets_data.csv, health_tips.csv, knowledge_base.csv) ไว้ในโฟลเดอร์ data/

รันไฟล์ embed_to_mongo.py:

ระบบจะดึงไฟล์ CSV ทั้งหมดมาอ่าน

หั่นข้อมูลที่ยาวเกินไป (Chunking) โดยใช้ RecursiveCharacterTextSplitter

แปลงตัวอักษรเป็น "ตัวเลขทางคณิตศาสตร์" (Vector Embeddings) โดยใช้โมเดล intfloat/multilingual-e5-base

ส่งข้อมูลทั้งหมด (พร้อม Metadata เช่น type, value) ไปเก็บไว้ในฐานข้อมูล MongoDB Atlas

จบเฟสแรก: บอทมึงมีความจำพร้อมดึงมาใช้งานแล้ว

🚀 ช่วงที่ 2: เมื่อยูสเซอร์เริ่มคุย (Runtime Phase)
นี่คือ Flow หลักที่เกิดขึ้นทุกครั้งที่มึงพิมพ์ข้อความผ่านหน้าเว็บ/แอป

จุดเริ่มต้น (Entry Point): ยูสเซอร์พิมพ์ข้อความส่งมาที่ API POST /api/chat ในไฟล์ app.py

ด่านหน้า (Fast Reply):

ข้อความวิ่งเข้าฟังก์ชัน get_demo_response() ก่อนเป็นอันดับแรก

เช็กว่ามียูสเซอร์พิมพ์คำสั้นๆ ลอยๆ เช่น "สวัสดี", "เครียด", "ดีใจ" ไหม

ถ้าเจอ: บอทจะหยิบคำตอบที่เตรียมไว้ (Hardcoded) ตอบกลับไปทันที (จบการทำงาน ไวโคตรๆ)

ถ้าไม่เจอ: ปล่อยข้อความให้วิ่งเข้า "สมองหลัก" (AI) ต่อไป

ด่านวิเคราะห์เจตนา (NLP Pipeline):

ข้อความถูกส่งไปที่ไฟล์ nlp_pipeline.py (ฟังก์ชัน analyze_user_input())

ใช้ pythainlp ตัดคำ หา Keywords

สกัดตัวเลข (Constraints) เผื่อยิงคำถามว่า "ขอเมนูไม่เกิน 500 แคล"

ประเมินเจตนา (Intent) ว่ายูสเซอร์อยาก "กิน" (Recommend_Food), อยาก "ออกกำลังกาย" (Calculate_Exercise) หรือ ถามหา "ความรู้" (Health_Advice)

ส่งข้อมูลพวกนี้กลับไปให้ app.py

ด่านค้นหาความจำ (RAG Service):

app.py โยน Intent, Keywords, Constraints ไปที่ไฟล์ rag_service.py (ฟังก์ชัน retrieve_top_k())

แปลงข้อความค้นหาเป็น Vector (ด้วย E5 โมเดลเดิม)

ไปควานหาข้อมูลที่คล้ายที่สุดใน MongoDB Atlas (มีการกรองด้วย pre_filter ตาม Intent เช่น หาเฉพาะหมวด food)

ดึงข้อมูลที่ดีที่สุดกลับมา (Retrieved Docs)

ด่านเครื่องคิดเลขส่วนตัว (Calculator Service):

ระบบเช็กว่าต้องคำนวณอะไรไหม

ถ้าถามเรื่องกิน: โยนข้อมูลน้ำหนัก ส่วนสูง ไปให้ calculator_service.py คำนวณหาค่า BMR ออกมาเป็นตัวเลข

ถ้าถามเรื่องออกกำลัง: เอาค่า MET จากข้อ 3 และระยะเวลาจากข้อ 2 ไปโยนให้ calculator_service.py คำนวณหาจำนวนแคลอรีที่เผาผลาญ

ได้ตัวเลขออกมาเตรียมส่งให้ AI

ด่านสรุปผลและสร้างคำตอบ (LLM Generation):

กลับมาที่ app.py ระบบจะเอาทุกอย่างมายำรวมกัน (ข้อมูลยูสเซอร์ + ประวัติแชทเก่า + ข้อมูลจากฐานข้อมูล RAG + ตัวเลขที่คำนวณได้)

จัดเรียงเป็น Prompt หรือ "คัมภีร์กฎเหล็ก" ใส่ลงไปให้โมเดลภาษา

ยิง Request ไปหา Ollama (Qwen2.5) ที่รันอยู่ในเครื่องมึง

Qwen อ่านข้อมูลทั้งหมด แล้วเรียบเรียงคำตอบเป็นภาษาไทยแบบสวยงาม เป็นธรรมชาติ

จุดสิ้นสุด (Output):

app.py รับคำตอบจาก Ollama (มีการหั่น Tag <think> ทิ้งให้เนียนๆ)

ส่งคำตอบนั้นกลับไปแสดงผลบนหน้าจอของยูสเซอร์
